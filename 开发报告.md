# mls-infer-opt 开发报告

把「LLM 推理引擎调优」组织成「模型训练」的自动化框架：给定 decoder-only 的模型结构信息，经
agent 多轮迭代，产出优化后的 `engine.py`，并留下 `runs/`、`results.log`、`output3.json` 供审计。

---

## 1. 目标与结构类比

输入是只读的模型结构（`model_config.json` + `weights/`），输出是一份自包含纯 PyTorch 的
`workspace/engine.py`，评测器按固定 API 契约调用：`create_engine` / `Engine.prefill / decode /
remove`（见 [generate/prompt.py](src/mls_infer_opt/generate/prompt.py#L28)）。

框架结构对应机器学习训练循环，每个调优动作映射到一个训练概念：

| 训练概念 | 本框架对应 | 代码位置 |
| --- | --- | --- |
| init | 装配上下文 + 落保守 baseline | [loop/\_\_main\_\_.py](src/mls_infer_opt/loop/__main__.py)、`bootstrap` |
| forward / eval | 评测一个候选（gate + bench） | [evaluate/](src/mls_infer_opt/evaluate/) |
| grad（梯度方向） | 分析反馈、定瓶颈、给下一步 | [analyze/](src/mls_infer_opt/analyze/) |
| train step | 生成一份新候选 engine | [generate/](src/mls_infer_opt/generate/) |
| loss / score | 三类吞吐的归一化几何平均 | [state/eval.py](src/mls_infer_opt/state/eval.py#L29) |
| keep-best / checkpoint | 发布更优候选为 best | `keep_best` / `finalize` |

主循环：init → forward(评测) → grad(分析) → step(生成候选) → 重新评测 → 若更优则发布为 best，
循环直到判停。

---

## 2. 分层架构（建设顺序 = 依赖顺序）

开发顺序为先定契约、再建能力层（agent 起步、逐阶段铺开）、最后由 loop 编排反馈循环，依赖单向
向下：

```
state/        契约层：跨阶段产出格式（最先搭，所有层共享）
searchspace/  搜索空间领域层：space + compat + Policy 聚合（依赖只向下到 state）
能力层（各阶段能力，agent 在需要的阶段接入）：
  llm/          agent：OpenAI Responses + function-tool + 读写（能力层起点）
  generate/     train step：产候选 engine（Policy + 开发规范 → engine.py）
  evaluate/     forward：正确性门 + 性能 benchmark（子进程隔离）
  analyze/      grad：诊断瓶颈 + 给下一个 Policy
loop/         反馈循环：确定性状态机，编排各能力 + 维护 LoopState + 发布 best
present/      产出层：实时进度 + results.log / 验收块
```

契约层（state）与产出层（present）是横向支撑；searchspace 是 analyze 与 generate 共享的搜索空间
领域模型（轴/选项/冲突消解/Policy 聚合），两者都向下依赖它、彼此不再横向 import；能力层是被编排
的各阶段，loop 专门负责把它们串成反馈循环。下文按此分层展开：第 3 节契约层，第 4 节逐个阐述能力层
各阶段（含各自的效果考量），第 5 节反馈循环，第 6 节产出。

---

## 3. 契约层：state

所有阶段之间交换的对象都是 [state/](src/mls_infer_opt/state/) 里的纯 dataclass，先于业务逻辑
搭建。契约按内存对象图设计——评测结果直接长在候选身上，不走关系库/外键/并行表（见记忆
`contracts-are-object-graphs`）：

- [`TaskContext`](src/mls_infer_opt/state/context.py)：一次会话的只读上下文（模型配置/设备/三类
  路径根/判停上限/环境快照）。派生路径用 `@property` 算出，约束 `engine.py` 只能写到约定位置。
- [`Policy`](src/mls_infer_opt/state/policy.py)：搜索空间里的一个点，analyze 产出、generate 消费的
  共享契约。类型落在最底层 state，analyze 与 generate 都能直接引用、互不横向 import（聚合逻辑在
  searchspace，见第 4 节）。
- [`Candidate`](src/mls_infer_opt/state/candidate.py)：一次 train step 的产物节点。诞生即定的不可变
  事实（id/kind/parent）+ 后填的评测结果（`gate`/`bench`），直接挂在对象上。源码不进内存、落盘到
  `candidates/{id}/engine.py`。
- [`GateResult` / `BenchResult`](src/mls_infer_opt/state/eval.py)：正确性与性能物理分离（硬布尔门
  vs 性能标量）。「只有过门候选才有性能」由 `Candidate.attach_bench` 在赋值处校验。
- [`LoopState`](src/mls_infer_opt/state/loop.py)：整个 run 唯一的主状态实例；候选构成内容寻址表，
  事件流 append-only。候选当前处于哪一步由 gate/bench/best 的存在性派生（`candidate_status`），不存
  冗余字段。

效果考量：契约先行让四个阶段可以独立开发、独立测试（hooks 可注入 fake，不必真跑 torch/LLM）；
对象图而非外键，避免全量候选常驻内存与灌爆 report JSON；状态派生而非冗余存储，消除不一致来源。

## 4. 能力层

能力层覆盖 agent、generate、evaluate、analyze 四个阶段。建设从 agent 起步、再逐阶段接入；阶段之间
不互相 import，只通过 state 契约与 searchspace 领域层交换/共享数据，由 loop 串联。各阶段连同自身
的效果考量分述如下。

### 4.1 agent（llm）

职责：[llm/](src/mls_infer_opt/llm/) 的 [`OpenAIAgentClient`](src/mls_infer_opt/llm/openai_client.py)
封装 OpenAI Responses API，支持本地 function-tool 执行循环（`run_agent`），并兼容单次
`generate(prompt)` 接口。它只提供调用、function call、读写工具，不含业务，由 generate / analyze
在需要 LLM 的阶段接入。

效果考量：

- **可选增益**：LLM 缺 key / 没装 SDK / 调用失败都返回不可用（`available=False`），由上层走规则
  兜底，框架在无 LLM 时仍能完整运行。
- **不假死**：客户端设 `max_retries=0`，关掉 SDK 默认的超时静默重试（最坏 ≈ timeout×3），重试交
  上层循环控制，避免长输出下放大成数分钟假死。
- **能力与业务解耦**：tool-loop 的每轮调用、工具执行、用量都有审计记录（`AgentResult` /
  `ToolCallRecord`），但客户端本身不知道任何业务语义。

### 4.2 generate（train step）

职责：[generate/](src/mls_infer_opt/generate/) 把 Policy + 开发规范渲成 prompt、产出一份完整
`engine.py` 候选。三个入口同一件事、条件不同：`bootstrap`（不依赖 LLM，落 pristine baseline，永久
兜底）、`propose`（按 Policy 产新候选）、`repair`（按结构化报错定向修复）。

关键抽象 —— Policy 与多维搜索空间（定义在 searchspace 领域层，generate 与 analyze 共享）：不让
agent 自由发挥，而是先把一次 forward 数据流经的层次拆成若干正交的轴，构成唯一的搜索空间真相源
[searchspace/space.py](src/mls_infer_opt/searchspace/space.py)——16 条轴、5 个组件分组
（cache / batching / operator / precision / weight_layout）。三个约定：

- **baseline-first**：每条轴 `options[0]` 等价于现有 baseline，「全默认轴」渲染出来就是 baseline
  本身，bootstrap 不需要特例。
- **轴名/选项名即契约**：同时是 policy 键、prompt 渲染、`strategy_tags`、analyze 引用策略的标识。
- **数值敏感标注**：🔴 改了会动 logits（可能顶破容差），🟢 正确实现即与 baseline 数学等价。

`Policy` 是在这些轴上各选一个选项加被激活轴的 knobs。从任意来源到一份合法 Policy 走固定四步管线
（[searchspace/policy.py](src/mls_infer_opt/searchspace/policy.py#L88) `aggregate`）：normalize（填默认、
丢非法轴/选项）→ resolve（消解轴间冲突）→ fill knobs（仅为被激活轴填 knob）→ Policy + strategy_tags。
generate 的输入即 Policy + 开发规范：[generate/prompt.py](src/mls_infer_opt/generate/prompt.py) 把
稳定的 `ENGINE_CONTRACT`（API 契约 / 自包含硬规则 / 正确性目标 / 输出格式）叠上 Policy 的非默认轴
中文指令、model_config、参考代码、analyze 方向，拼成 prompt。

效果考量：

- **内层自检收敛**：generate 不是调一次 LLM，而是确定性的自检重试自闭环
  （[codegen.py:135](src/mls_infer_opt/generate/codegen.py#L135) `_generate`）：出码 → 静态自包含
  早筛（语法/import 白名单/API arity）→ quick gate（子进程隔离、对 oracle 比 logits）→ 不过就把
  结构化错误回灌、转 repair 提示让模型修自己的码，循环到过门或耗自检预算（上限 4 轮）。控制权
  全程在确定性代码里，不依赖模型自觉调工具。
- **只产候选、无发布权**：内层 quick 自检是 ephemeral、不进 state；正确性由外层 evaluate 权威保证。
  LLM 不可用 / 失败 / 产垃圾 / 自检始终不过都返回 `None`（loop 当作这轮没收益走兜底），不抛异常。
- **冲突消解可复现**：[searchspace/compat.py](src/mls_infer_opt/searchspace/compat.py) 对「合批解码需 KV 缓存」
  「enable_gqa 需 SDPA」这类依赖采用依赖方让步——违反时把更激进的依赖轴退回 baseline 默认。退回
  默认不引入新风险，且对同一输入决定性。
- **内容寻址判重**：候选 id = `r{round}-{sha1(code)[:8]}`
  （[candidate.py:37](src/mls_infer_opt/state/candidate.py#L37)），可 O(1) 判「这段代码是否已测过」
  从而跳过重复评测；policy 序列化 `sort_keys` 字节级可复现，便于审计 diff。

### 4.3 evaluate（forward）

职责：[evaluate/](src/mls_infer_opt/evaluate/) 给候选两层信号——正确性硬门 `GateResult`（syntax →
api → correctness 三阶段）与性能 `BenchResult`（prefill/decode/mixed 三类吞吐 + 峰值显存）。只有过门
候选才进入性能评测。决定性、不调 LLM。

效果考量：

- **外层 full gate 是唯一权威**：挂到 `candidate.gate` 的只有 loop 在 keep-best/发布前重跑的 full
  gate，不采用 generate 内层 quick 自检的 ephemeral 结果，也不信候选自报。正确性判定为对官方
  reference model 比 `allclose(atol=1e-2, rtol=1e-2)`（[gate.py](src/mls_infer_opt/evaluate/gate.py)），
  逐 case fail-fast。oracle 与候选喂完全相同的确定性事件流（同 seed 生成
  [cases.py](src/mls_infer_opt/evaluate/cases.py)），否则比对无意义。
- **子进程隔离**：所有 torch / 风险代码只在 `evaluate.worker` 子进程跑，父进程纯 stdlib、零 torch。
  坏候选崩溃/超时只死子进程，[runner.py](src/mls_infer_opt/evaluate/runner.py) 把超时→杀进程、
  非零退出/非法 stdout 一律翻成结构化失败 gate，上层始终拿到结果，不抛异常。
- **泛化验证**：隐藏评测会变 batch / 长度 / 步数 / 顺序，因此两侧都防过拟合。正确性侧除照搬的 6 类
  标准 case，额外加变 batch/长度/顺序的泛化抽测（`general_prefill`/`general_decode`，
  [cases.py:81](src/mls_infer_opt/evaluate/cases.py#L81)）；性能侧 full 工况设为不规则（ragged）——
  decode/mixed 每请求长度与停止步各异、跨多个 seed（7/101/202），逐 seed 取下半均值聚合
  （[bench.py](src/mls_infer_opt/evaluate/bench.py)）。等长 lockstep 流会被分组策略过拟合，ragged 流
  更能反映 varlen 批处理 / KV-cache 的鲁棒性；跨 seed 的 decode tps 离散度回喂 analyze 作抗不规则
  信号。
- **score 口径**：[`geomean_score`](src/mls_infer_opt/state/eval.py#L29) 用三类吞吐比值的加权几何
  平均（decode 0.60 / mixed 0.25 / prefill 0.15）：scale-free，不被某一类的量级绑架；任一类塌到 0
  即把整体拉到近 0，对应该类失败即灾难。worker 侧只算临时自评（参照 ref=1），loop 内的权威归一化
  在 loop 层完成（见 5 节）。

### 4.4 analyze（grad）

职责：[analyze/](src/mls_infer_opt/analyze/) 每轮看反馈，定位瓶颈，从当前 best 出发给下一步方向或
判停。流程（[grad.py](src/mls_infer_opt/analyze/grad.py)）：先把 `LoopState` 汇总成 ephemeral 的
`Situation`，再做①确定性硬判停（预算/轮数/连续无提升）②LLM 要方向（单次调用 + 确定性解析）③不可用
就退回 rule-based 贪心阶梯。

关键抽象 —— 基于 best 叠 delta（梯度叠加）：每轮 propose 的父代是当前 best 的源码加 best 的 Policy，
analyze 只产相对 best 要改动的轴（`axes_delta`），经 searchspace 的 `merge(best_policy, axes_delta=…)`
得到下一个完整 Policy（[grad.py:80](src/mls_infer_opt/analyze/grad.py#L80)）。等价于把梯度叠加到当前
参数上做局部搜索：复用已验证的稳收益，避免每轮从零重新踩坑。叠加用的是共享的 searchspace 领域层
（merge/aggregate/compat），analyze 不横向 import generate——这正是把搜索空间抽成独立层的目的。

效果考量：

- **LLM 不可靠时仍给合法方向**：rule-based 兜底是一条贪心阶梯
  （[heuristic.py](src/mls_infer_opt/analyze/heuristic.py#L71) `MOVES`），按优化主线先验排序：KV
  增量缓存（收益最大）→ RoPE 预算表 → batched prefill → 合批解码 → SDPA → GQA → 权重融合 → bf16，
  前置依赖在前、低风险(🟢)优先、数值敏感(🔴)靠后。相同 best 给相同下一步，可复现可测。
- **never-throw**：内部任何异常（含建态势、构 Policy）都翻成一条 error 事件并返回 `None`（= 停，
  best 已是安全产物）。
- **无发布权、不写 stop_reason**：analyze 每轮只 emit 一条 `analyze` 事件，停因放进事件 `data`，由
  loop 读后落到 `LoopState.stop_reason`（见记忆 `analyze-record-via-events`）；knob 只进 Policy.knobs，
  绝不碰 model_config 的结构字段。

## 5. 反馈循环：loop

职责：[loop/trainer.py](src/mls_infer_opt/loop/trainer.py) 的 `run_loop` 是外层确定性控制器，不问
LLM 做业务判断、不信候选自证、不产 engine 代码，只负责编排各能力、维护唯一的 `LoopState`、发布
已验证 best。

```
bootstrap → 评测 → keep-best（冻结 baseline_score 作 speedup 锚点）
  └─ while best 存在 且 未判停：
       analyze(state)  ──► 下一个 Policy（带 rationale）或 None(停)
       propose(Policy, best_code) ──► 新 Candidate（基于当前 best）
       evaluate(Candidate) ──► 填 gate / bench
       keep_best?  ── 是 ─► 增量发布 engine.py + output3.json
                   └─ 否，且 gate 不过 ─► repair 重试（外层）
finalize → final gate 复核 → 权威终发
```

一轮的实现见 `_run_policy_round`（[trainer.py:362](src/mls_infer_opt/loop/trainer.py#L362)）。

效果考量：

- **发布以正确性为硬门**：只有 `gate.passed` 且分数严格更高的候选才提升为 best（`keep_best`），
  `set_best` 在赋值处再校验一遍；只有当前 best 能被发布。
- **score 归一化单一咽喉点**：父进程统一按 baseline per-category tps 把候选 score 归一化为加速比
  （`_normalize_score`，[trainer.py:463](src/mls_infer_opt/loop/trainer.py#L463)），是所有 score
  消费者（keep-best 比较 / speedup 展示 / analyze）之前唯一的归一化处，口径一致且诚实。
- **抗中途 kill**：评测有约 30min 外墙，`run.sh` 设 28min 内部硬墙（`MLS_TIME_BUDGET_S=1680`），
  loop 到点后在下一轮开始前停 + finalize。每刷新一次 best 立刻把已过 gate 的 `engine.py` 拷到发布点
  并同步刷新 `output3.json`（[trainer.py:249](src/mls_infer_opt/loop/trainer.py#L249)），`results.log`
  逐条 append。任意时刻被 kill，盘上都是当前最优。
- **三级兜底链**（[loop/\_\_main\_\_.py](src/mls_infer_opt/loop/__main__.py)）：最优正确候选 → 已验证
  baseline → 原始 pristine baseline。最外层 try/finally 保证两条对外契约即便 run_loop 崩溃也成立：
  始终 exit 0，`workspace/engine.py` 必在盘上。

## 6. 产出与可审计性

| 产物 | 内容 | 写入时机 |
| --- | --- | --- |
| `workspace/engine.py` | 唯一发布出口（已验证 best） | keep-best 增量发布 + finalize 终发 |
| `workspace/output3.json` | 摘要 + `result` 判定 + `rounds[]` 逐轮推理（best/score/speedup/正确性/停因/诊断→策略→评测→结论） | 随 best 同步刷新 + finalize |
| `runs/{run_id}/report.json` | 任务结果记录（内容同 output3.json） | finalize |
| `runs/{run_id}/final/results.log` | 逐条结构化事件日志（增量写、抗中途 kill） | 逐条实时 append + finalize 全量 |
| `runs/{run_id}/candidates/{id}/` | 每个候选的 `engine.py` + `policy.json` | 落盘即留 |
| `runs/{run_id}/final/` | 本次 run 终态留档：`engine.py` / `output3.json` / `results.log`（供复盘） | finalize |

> `report3.*` 是本开发报告这类**人手写**文档，运行时不自动产出；逐轮推理已机读化进 `output3.json` 的 `rounds[]`。

[present.py](src/mls_infer_opt/present.py) 把每条 `AgentEvent` 渲染成可 grep 的表头 + 缩进数据块，
`analyze → generate → evaluate → keep_best` 四类事件按诊断 → 策略 → 评测 → 结论的顺序排列，逐轮
可观测。

---

## 7. 小结

各层职责单一、依赖单向：state 定契约，能力层（agent/generate/evaluate/analyze）各管一段并自带
对应的正确性与鲁棒性保障，loop 串成反馈循环并独占发布权与兜底责任。LLM 全程是可选增益，缺失或
失败时各层退回确定性路径，任意失败或中断下都满足始终 exit 0、engine.py 必在盘、产物反映当前最优。
